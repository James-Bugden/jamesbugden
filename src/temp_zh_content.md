## 前言：為什麼大多數求職者在起跑點就輸了

我審閱過超過 **20,000 份履歷。** 我在 **Uber** 和 **Netskope** 等公司錄用了超過 **500 人。**

被錄取的求職者和沒被錄取的求職者，最大的差別是什麼？不是經驗。不是學歷。**是他們的思考方式。**

大多數求職者走進面試就開始即興發揮。他們聽到問題，就脫口說出腦海中第一個想到的東西。他們東拉西扯，希望能碰巧說中什麼。

最好的求職者做的事不一樣。他們暫停。他們拆解問題。他們用有結構的方式大聲思考。然後才回答。

這種結構化的思考方式有個名字。它來自渡邊健介的 **《解決問題最簡單的方法》**（英文版：Problem Solving 101）。渡邊健介是前麥肯錫顧問，後來離開去教日本孩子如何解決問題。他的書成為**日本商業類暢銷書第一名。**

這本書中的工具，和顧問、商業領袖、以及我面試過最強的求職者所使用的工具完全一樣。

這份指南教你這些工具。你會學到如何在每一場面試中運用它們。你回答問題時會比 **95% 的其他求職者**更清楚、更有結構、更有信心。

不需要顧問背景。不需要 MBA。這些是簡單的工具。任何人都能學會。

> **以我錄用 500+ 人的經驗來說：** 當我面對一位用結構化方式思考的求職者，我在前 5 分鐘內就會注意到。他們之所以突出，是因為大多數人從來沒學過這件事。你即將學會。

### 這份指南包含什麼

這份指南很長。你不需要一次讀完。以下是每個章節的內容，讓你知道該往哪裡看。

## 五種面試者類型（其中四種不會被錄取）

渡邊健介在書中介紹了五種角色類型。每一種代表一種不同的解決問題方式。我每個禮拜都會在面試中看到這五種人。

```
  五種面試者類型
  ==============

  冷凍人        「我不知道……」              --> 未錄取
  抱怨者        「我老闆很糟糕……」          --> 未錄取
  夢想家        「我想要創造影響力……」      --> 未錄取
  衝衝衝        「所以我馬上就做了……」      --> 未錄取
  問題解決者    「首先，我找出了              --> 錄取
                 根本原因……」
```

以下是從書中翻譯到面試間的版本。

### 1. 冷凍人（書中的「藉口小子」）

在書中，藉口小子一面對挑戰就放棄。她說「我永遠做不到」然後就停下來不嘗試了。

**在面試中，** 這是那種聽到困難問題就當機的求職者。他們說「我不知道」然後跳過。他們只給一句話的答案。他們迴避任何感覺有風險的東西。

**面試官想的是：** 「這個人在壓力下會崩潰。」

**冷凍人的回答聽起來像這樣：**

*面試官：「跟我說一個你面對模糊情境的經驗。」*

*「嗯，我不太確定那個我有沒有好的例子。我是說，我上一份工作通常事情都蠻清楚的。我想有時候事情不太明確，但我想不起來具體的例子。抱歉。」*

這個回答讓評估結束了。面試官直接跳過你。你現在落後每一個其他求職者。

### 2. 抱怨者（書中的「批評小子」）

在書中，批評小子指出每個人做錯的事，但他自己從來不做任何事。

**在面試中，** 這是那種整場面試都在講現職問題的求職者。爛主管。爛文化。爛系統。他們有一長串的抱怨，但**零個解決方案。**

**面試官想的是：** 「這個人到了這裡，事情出錯的時候也會怪所有其他人。」

**抱怨者的回答聽起來像這樣：**

*面試官：「你為什麼想離開現在的職位？」*

*「老實說，領導層沒有願景。我的主管不給明確方向，然後專案落後就怪我們。流程都是壞的。我已經告訴他們一百次需要改什麼，但沒有人聽。我需要一個重視好想法的地方。」*

每一句都在指著別人。沒有一句話描述這個求職者做了什麼來解決問題。

### 3. 夢想家（書中的「空想小子」）

在書中，空想小子有很大的想法但從不計畫如何實現它們。她滿足於只是想著她的夢想。

**在面試中，** 這是那種談論願景和策略但沒有任何具體內容的求職者。他們說「我想打造有意義的東西」或「我看到自己帶領一個團隊」之類的話。但當你問**怎麼做，**他們什麼都沒有。

**面試官想的是：** 「很棒的想法。零執行力。」

**夢想家的回答聽起來像這樣：**

*面試官：「你在前 90 天會做什麼？」*

*「我會先想了解大方向。然後我會尋找增加價值的方式，並在組織中建立關係。我覺得最重要的是帶來全新的觀點，串連其他人遺漏的點。」*

聽起來很聰明。什麼都沒說。沒有具體內容。沒有計畫。沒有可衡量的行動。

### 4. 衝衝衝（書中的「衝動小子」）

這一種會讓人驚訝。在書中，衝動小子不是懶惰。他很快跳入行動。他很努力工作。但他**從不停下來想一想**自己是否跑對方向。

**在面試中，** 這是那種直接跳到「這是我做的事」卻不解釋**為什麼**這樣做的求職者。他們跳過思考。跳過診斷。直接進入行動。

**面試官想的是：** 「這個人會把事情做完，但他們知道為什麼嗎？他們會解決對的問題還是看起來很忙？」

**衝衝衝的回答聽起來像這樣：**

*面試官：「跟我說一個你解決困難問題的經驗。」*

*「我們的業績下降了，所以我馬上開始打更多電話。我把外展量增加了一倍。我每天加班到很晚。我早上六點就寄追蹤信。最後業績回來了，因為我比問題更努力。」*

很多努力。沒有診斷。沒有根本原因。沒有策略。業績大概是因為完全不同的原因回升。

### 5. 問題解決者（書中的「解決問題小子」）

在書中，解決問題小子做了其他人都沒做的事。他們**先了解狀況。**他們**找到根本原因。**他們**制定計畫。**他們**執行。**然後他們**檢查結果並調整。**

**在面試中，** 這是那種回答前先問釐清問題的求職者。把問題拆成幾個部分的人。一步一步解釋自己推理過程的人。用真實數字分享結果的人。告訴你他們學到什麼的人。

**這就是我每次都會推薦錄取的求職者。**

**問題解決者的回答聽起來像這樣：**

*面試官：「跟我說一個你解決困難問題的經驗。」*

*「我們的業績在第三季下降了 20%。在做任何事之前，我把數據按區域、產品線和業務代表拆開來看。結果三個區域中有兩個沒問題。整個下降都來自東南區，在我們改變那裡的定價模型之後。」*

*「我訪談了那個區域的五位業務代表和三位停止購買的客戶。根本原因不是努力不夠或競爭。客戶說新的定價層級太令人困惑，他們搞不清楚該買哪個方案。」*

*「我和產品團隊合作，把六個選項簡化為三個。我們也為業務代表製作了一頁的比較表。六週內，東南區的業績恢復到上一季的 95%。」*

那個回答有診斷、根本原因、計畫、執行和帶數字的結果。那就是拿到錄取通知的回答。

```
  同一個問題，五種不同的回答：
  「跟我說一個你解決困難問題的經驗。」

  冷凍人：    「我想不到一個。」
  抱怨者：    「沒有人給我資源來解決它。」
  夢想家：    「我構想了一個更好的系統。」
  衝衝衝：    「我更努力工作直到解決了。」
  問題解決者：「我診斷了根本原因，制定了計畫，
               執行它，然後衡量了結果。」
```

> **以我的經驗來說：** 你大概是這些類型的混合。這是正常的。目標是在你給出的每一個回答中，讓自己更接近**第五型。**這份指南的其餘部分會告訴你具體怎麼做。

## 贏得面試的四步驟框架

渡邊健介整本書歸結為**四個步驟。**他教的每一個工具都支持這個流程：

```
  四步驟問題解決框架
  ==================

  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │   步驟一    │    │   步驟二    │    │   步驟三    │    │   步驟四    │
  │             │    │             │    │             │    │             │
  │   了解      │───>│   找出      │───>│   制定      │───>│   執行      │
  │   現況      │    │  根本原因   │    │  行動計畫   │    │  監控調整   │
  │             │    │             │    │             │    │             │
  └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘

  面試中的       「讓我確認我      「我把問題拆     「以下是我      「結果是 X。
  應用：          理解你問的        開，找到了       做的事，以      以下是我會
                  問題。」          真正的問題。」   及為什麼選      做得不同的
                                                     這個方法。」    地方。」
```

就是這樣。四個步驟。同樣的四個步驟適用於修復搖滾樂團的觀眾問題、存錢買電腦、或挑選巴西的學校。

它們也適用於**回答任何面試問題。**

以下是方法。

### 步驟一：了解狀況（先別回答）

當面試官問你問題時，大多數求職者馬上就開始說話。問題解決者做相反的事。他們**先確認自己理解了這個問題。**

這意味著：
- **暫停** 2 到 3 秒。整理你的想法。
- **問一個釐清的問題，**如果問題不夠明確。
- **用自己的話重述問題。**

在渡邊健介的書中，蘑菇戀人樂團只有 **15 個人**來看他們的演唱會。他們很想直接猜原因。但他們先列出了所有可能的原因再行動。（你會在後面的蘑菇戀人方法章節看到完整故事。）

**範例：步驟一的實際應用**

*面試官：「你會怎麼改善我們的客戶留存率？」*

*「在我回答之前，我想確認我用對的方式思考這個問題。你說的留存率，是指讓客戶在第一次購買後繼續留下，還是降低長期客戶的流失率？你是否知道最大的流失發生在哪裡？」*

那 10 秒的釐清讓面試官看到你在行動前會先思考。它也給你更好的資訊來作答。

### 步驟二：找出根本原因（挖得比表面更深）

大多數求職者給出表面層次的答案。「業績下降了，所以我打更多電話。」那是**衝動小子的回答。**你跳過了診斷。

蘑菇戀人假設沒有人知道他們的演唱會。問卷顯示**30% 的人已經知道了。**真正的問題是知道的人不想來。根本原因和他們的假設不同。

**範例：步驟二的實際應用**

*面試官：「跟我說一個你找到問題根本原因的經驗。」*

*「我們的客服工單量在一個月內翻倍了。表面的解釋是『更多客戶，更多工單。』但我拉出數據，按類型分類工單。80% 的增長來自一個類別：帳單問題。我進一步深入。我們三週前推出了新的定價頁面。那個頁面有一個格式錯誤，顯示了錯誤的年費方案價格。客戶按一個價格註冊，卻被收了另一個價格。根本原因不是更多客戶。是壞掉的定價頁面。」*

### 步驟三：制定行動計畫（展示你的思考過程）

找到根本原因後，解釋你打算怎麼處理它。這是面試官看到你策略思考的地方。

蘑菇戀人列出了 **13 個不同的想法**來提升出席率。他們按**影響力 vs. 執行難度**繪製每一個。然後挑選了最佳組合。在面試中展示同樣的結構化優先排序。

**範例：步驟三的實際應用**

*「一旦我確認根本原因是定價頁面的錯誤，我有三個選項。選項 A：立即恢復到舊的定價頁面。快速修復，但我們會失去新的設計改進。選項 B：修復新頁面上的格式錯誤。最好的長期方案，但需要開發團隊的幫助，而他們在衝刺期。選項 C：在等待開發修復的同時，加一個臨時橫幅說明正確價格。」*

*「我選了組合方案。我推動開發團隊做緊急修復，因為這是影響營收的 bug。同時，我加了一個臨時橫幅，讓客戶在修復上線前的 24 小時內不會被搞混。」*

### 步驟四：執行並調整（分享結果和教訓）

這是你畫上句號的地方。**分享發生了什麼。**盡量用真實數字。然後**分享你學到了什麼。**

蘑菇戀人追蹤了一切。他們的第四場演唱會吸引了 **200 人，**從 15 人增加。他們知道確切的數字：知名度從 30% 上升到 90%，出席率從 10% 上升到 50%。在你的回答中帶來同樣的具體程度。

**範例：步驟四的實際應用**

*「開發團隊在 24 小時內修好了頁面。客服工單一週內恢復正常。我們也退款給 340 位被多收費的客戶，這讓那群客戶的留存率比我們的平均值高了 15%。他們很感謝我們發現了錯誤並做出補救。」*

*「我從中學到的教訓是：每次我們更改面向客戶的頁面，現在都會專門針對定價準確性做 QA 檢查。那一個流程改變，之後已經防止了兩次類似的錯誤。」*

> **以我錄用 500+ 人的經驗來說：** 這個四步驟框架對應到大多數面試教練教的 **STAR 方法**（情境、任務、行動、結果）。但它走得更遠。STAR 方法告訴你要**包含什麼。**渡邊健介的框架教你**如何思考**每一個部分。這就是背誦式答案和真正強大答案之間的差別。

## 分解樹：如何拆解任何面試問題

**分解樹**是一個視覺工具，幫助你把大問題拆成小塊。你從左邊一個問題開始，往右邊分出分支，從寬泛到具體。

渡邊健介用了一個簡單的例子：「你如何增加胡椒罐搖出來的胡椒量？」

```
  分解樹：胡椒罐
  ================

  如何讓一次搖        ┌─── 增加頂面的面積
  出更多胡椒？  ─────┤
                      ├─── 增加表面積搖出    ┌── 每個面積更多孔
                      │    的胡椒量      ─────┤
                      │                       └── 更大的孔
                      │
                      └─── 讓胡椒更容易   ─────── 更小的胡椒粒
                           出來
```

一個問題產生四個不同的解決方案。「搖更大力」根本不在清單上。這就是分解樹的力量。

### 如何在面試中使用分解樹

大多數面試中你不會在白板上畫樹。但你會**用同樣的思考模式大聲說出來。**

當面試官問**「你會如何改善 X？」**時，這樣做：

1. **把問題拆成** 2 到 3 個主要類別。
2. **在每個類別下，**列出 1 到 2 個具體想法。
3. **解釋**你會先處理哪個分支以及為什麼。

**範例：「你會如何改善我們的新人到職流程？」的分解樹**

```
  分解樹：改善新人到職
  ======================

  如何改善        ┌── 到職前溝通     ┌── 更早寄送歡迎包
  到職流程？ ─────┤              ────┤
                  │                  └── 第一天前就指派夥伴
                  │
                  ├── 第一週體驗     ┌── 有結構的每日行程
                  │            ─────┤
                  │                  └── 減少文書，多些人際互動
                  │
                  └── 前 90 天       ┌── 每週主管一對一
                      支援     ─────┤
                                    └── 30/60/90 天里程碑
```

**使用這棵樹的範例回答：**

*「我會用三個面向來思考這件事。第一，**到職前溝通。**新人在第一天之前是否有得到足夠的資訊？像是歡迎包、工具權限、指派的夥伴。第二，**第一週體驗。**行程是有結構的還是混亂的？大多數公司在第一週塞滿文書，卻跳過關係建立。第三，**前 90 天的持續支援。**有沒有夥伴制度或主管的定期確認？」*

*「以我看到的情況，最大的流失發生在第二個面向。新人在前 5 天就形成了對公司的看法。我會從建立一個有結構的第一週行程開始，平衡到職任務和認識團隊。」*

**另一個範例：「你會如何增加這個產品的營收？」**

```
  分解樹：增加營收
  ==================

  如何增加        ┌── 獲得更多        ┌── 改善行銷
  營收？    ─────┤    客戶      ─────┤
                  │                   └── 改善轉換率
                  │
                  ├── 提高每位客戶    ┌── 提高價格
                  │   的營收    ─────┤
                  │                   └── 追加銷售/交叉銷售
                  │
                  └── 留住客戶        ┌── 降低流失率
                      更久      ─────┤
                                     └── 增加合約期限
```

*「我會把營收成長拆成三個面向。獲得更多客戶、從每位客戶身上獲得更多營收、或讓客戶留更久。每一個都有不同的操作方式。在選定方向之前，我想先知道最大的缺口在哪裡。如果你的轉換率是 1% 但留存率是 95%，我會專注在轉換。如果留存率是 70%，那就是錢正在流失的地方。」*

### 什麼時候使用分解樹

- 「你會如何處理這個問題？」
- 「你在前 90 天會做什麼？」
- 「你會如何增加這個產品的營收？」
- 「你會考慮哪些因素？」
- **任何要求你解決、改善或規劃某件事的問題。**

### 各職位的分解樹

**工程：「你會如何降低頁面載入時間？」**

*「我會拆成三個層。第一，**網路層。**我們有沒有優化資產傳遞？CDN、壓縮、快取標頭。第二，**後端層。**API 呼叫是否緩慢？資料庫查詢、伺服器回應時間、傳送了不必要的數據。第三，**前端層。**我們的渲染效率高嗎？打包大小、延遲載入、阻塞腳本。」*

*「我會先測量每一層。如果後端在 100 毫秒內回應，但前端需要 3 秒來渲染，我會專注在前端。如果後端需要 2 秒，那就從那裡開始。」*

**營運：「你會如何減少出貨錯誤？」**

*「我會把出貨錯誤拆成三個類別。第一，**揀貨錯誤。**從架上拿了錯誤的商品。第二，**包裝錯誤。**拿對了商品但數量錯誤或貼了錯誤的地址標籤。第三，**系統錯誤。**訂單管理系統傳送了不正確的資訊到倉庫。」*

*「每一種有不同的修復方式。揀貨錯誤需要更好的倉庫佈局或條碼掃描。包裝錯誤需要一個驗證步驟。系統錯誤需要數據完整性稽核。我會拉出過去 3 個月的錯誤報告，分類它們，然後先專注在最大的桶子。」*

**客戶服務：「你會如何減少重複來電？」**

*「我會把重複來電分成三個桶子。第一，**未解決的問題。**客戶回電是因為第一次沒解決他們的問題。第二，**溝通不清楚。**問題解決了，但客戶不了解結果或下一步。第三，**反覆發生的問題。**同一個問題不斷發生在同一個客戶身上。」*

*「每個桶子需要不同的解決方案。未解決的問題需要更好的首次解決率培訓。溝通不清楚需要更好的結案話術。反覆發生的問題需要產品或流程的修復，而不是客服的修復。」*

**業務：「你會如何縮短銷售週期？」**

*「我會把銷售週期按階段拆開，找出哪一個耗時最長。第一，**從潛在客戶到第一次會議。**外展速度慢還是回覆率低？第二，**從第一次會議到提案。**我們是否花太長時間來確認需求或建立商業論據？第三，**從提案到成交。**交易是否卡在談判、採購或法務審查？」*

*「以我的經驗，最大的時間消耗通常在提案到成交之間。交易卡在等待客戶端的內部審批。我會專注在給支持者內部銷售的工具：ROI 計算器、高層摘要和類似公司的參考電話。」*

**人員管理者：「你會如何提升團隊生產力？」**

*「我會把生產力拆成三個領域。第一，**清晰度。**每個人都知道自己該做什麼以及為什麼重要嗎？第二，**容量。**工作量合理嗎，還是人們分散在太多專案上？第三，**能力。**人們有做好工作所需的技能和工具嗎？」*

*「大多數主管直接跳到能力。他們假設人們需要培訓。以我的經驗，問題通常在清晰度或容量。當人們知道優先事項並有時間專注在上面時，生產力自然提升。」*

> **以我的經驗來說：** 當求職者在回答前先把問題分成幾個類別，我會在筆記上寫下**「結構化思考者」。**單單那個標籤就能讓求職者在流程中向前推進。那是招募官能給的最高讚美之一。

## 是非樹：大聲診斷問題

**是非樹**根據是或否的問題，把人或事物分到不同的桶子裡。每個答案通向一個桶子（解釋）或另一個問題。

在書中，蘑菇戀人用是非樹來分類學校裡的人：

```
  是非樹：蘑菇戀人演唱會出席
  ============================

  全校 500 位
  師生
       │
       ▼
  他們知道演唱         否 ───> 350 人 (70%)
  會的事嗎？ ──────────┐       「不知道」
                       │
                      是
                       │
                       ▼
  他們有來過           否 ───> 135 人 (27%)
  嗎？ ───────────────┐       「知道但從沒來過」
                       │
                      是
                       │
                       ▼
  他們固定來           否 ───>   3 人 (1%)
  嗎？ ───────────────┐       「來過一次就不來了」
                       │
                      是
                       │
                       ▼
                      12 人 (2%)
                      「忠實粉絲」
```

這幫助他們精確定位問題出在哪裡。不是知名度（他們假設的）。**是轉換率**（真正的問題）。知道演唱會的人中有 90% 從來沒去過。那才是要修的桶子。

### 如何在面試中使用是非樹

是非樹適合**行為面試問題。**「跟我說一個你解決問題的經驗。」不要講一個雜亂的故事，而是帶著面試官走過你的診斷過程。

**範例：「跟我說一個你找出並解決問題的經驗。」**

```
  是非樹：客戶流失診斷
  ======================

  流失率從 5%
  飆升到 12%
       │
       ▼
  是在免費試用階段       是 ──> 不是問題。
  還是付費階段流失？─────┐      試用轉換穩定。
                         │
                    付費階段
                         │
                         ▼
  是在第一個月還是       是 ──> 根本原因區域
  之後取消？ ───────────┐      第一個月體驗
                         │
                    第一個月
                         │
                         ▼
  取消前有聯繫客服       是 ──> 不是問題。
  嗎？ ─────────────────┐      只有 10% 聯繫客服。
                         │
                    無聲離開
                         │
                         ▼
                    根本原因：
                    新的付費客戶不知
                    道如何快速獲得價
                    值。他們在求助之
                    前就離開了。
```

**完整的範例回答：**

*「我們的客戶流失率在一個季度內從 **5% 飆升到 12%。**我需要搞清楚我們在哪裡流失客戶。」*

*「第一個問題：我們是在免費試用階段流失客戶，還是在他們成為付費客戶之後？我查了數據。**答案：付費客戶。**試用轉換是穩定的。所以問題不在獲客。」*

*「下一個：他們是在第一個月內取消還是幾個月後？**答案：第一個月。**所以問題在早期體驗，不是長期價值。」*

*「下一個：他們取消前有聯繫客服嗎，還是無聲離開？**答案：90% 無聲離開。**他們甚至沒有嘗試尋求幫助。」*

*「這告訴我根本原因是**新付費客戶的到職體驗。**他們不知道如何快速從產品中獲得價值。所以我建立了一個 7 天到職電子郵件序列，配合引導式設定清單。第一個月的流失率在兩個月內從 12% 降回 6%。」*

**另一個範例：「帶我走過你排查技術問題的過程。」**

*「我們的應用程式載入時間一夜之間翻倍了。我需要快速找到原因。」*

*「它影響所有使用者還是一部分？**所有使用者。**所以不是特定裝置。它影響所有頁面還是特定頁面？**所有頁面。**所以不是單一功能。它從特定時間開始嗎？**是的，週二凌晨 2 點。**週二凌晨 2 點部署了什麼？**一個資料庫遷移。**那就是根本原因。遷移加了一個索引，本應加速查詢，但在每次查詢時都執行全表掃描。」*

*「我們回滾了遷移，載入時間恢復正常，資料庫團隊重新設計了索引後再重新部署。」*

### 什麼時候使用是非樹

- 「跟我說一個你找出問題的經驗。」
- 「你是怎麼發現問題出在哪裡的？」
- 「帶我走過你的排查流程。」
- **任何需要你展示如何診斷問題的問題。**

### 各職位的是非樹

**工程：「跟我說一個你除錯正式環境問題的經驗。」**

*「我們的付款處理有大約 5% 的交易開始失敗。我需要快速隔離原因。」*

*「第一：是所有付款方式都受影響還是一種？**一種。只有信用卡。**簽帳卡和 PayPal 沒問題。下一個：是所有信用卡還是特定卡種？**只有 Visa。**Mastercard 和 Amex 處理正常。下一個：是所有 Visa 交易還是特定區域？**只有帳單地址在美國以外的交易。**」*

*「根本原因：我們的付款閘道推送了一個更新，改變了 Visa 國際地址驗證的方式。新的驗證拒絕了沒有郵遞區號的地址，而很多國際地址沒有郵遞區號。我和閘道供應商協調回滾驗證變更，並修補了我們的整合，在郵遞區號欄位為空時傳送一個預設值。失敗率在 24 小時內從 5% 降回 0.1%。」*

**營運：「跟我說一個你找到品質問題來源的經驗。」**

*「我們的產品退貨率在一個月內從 2% 飆升到 8%。我需要搞清楚什麼改變了。」*

*「第一：是所有產品還是特定 SKU？**三個 SKU。**都在同一個產品線。下一個：退貨來自所有倉庫還是一個？**一個倉庫。我們的達拉斯設施。**下一個：是所有班次還是特定班次？**只有夜班。**」*

*「我去了達拉斯設施觀察夜班。結果兩週前安裝了一台新的包裝機。它校準不正確，封箱太鬆。產品到貨時已經損壞。根本原因不是產品。是一個倉庫裡一個班次的一台機器。我們重新校準了機器，那些 SKU 的退貨率在兩週內降回 1.5%。」*

**客戶服務：「跟我說一個你找出客戶滿意度下降原因的經驗。」**

*「我們的 CSAT 分數在兩個月內從 87% 降到 71%。我需要找出原因。」*

*「第一：是所有渠道都下降還是一個？**電話穩定。線上客服下降了。**下一個：是所有問題類型都下降還是特定的？**只有帳務相關的線上客服。**下一個：是否和任何變更同時發生？**是的。我們兩個月前推出了新的帳務聊天機器人。**」*

*「我檢視了 50 份 CSAT 低分的線上客服記錄。聊天機器人在每次帳務對話的前 3 分鐘處理，然後才轉給真人。但它要求客戶重新輸入他們已經打過的資訊。客戶在接觸到真人客服之前就已經感到沮喪了。我和產品團隊合作，把聊天機器人的對話歷史傳給客服。帳務線上客服的 CSAT 在一個月內恢復到 83%。」*

**業務：「跟我說一個你找出交易停滯原因的經驗。」**

*「我的銷售管道中有 200 萬美元的交易已經在『提案已發送』階段停了超過 30 天。我需要理解瓶頸。」*

*「第一：是所有交易規模都停滯還是特定的？**只有超過 5 萬美元的交易。**較小的交易正常成交。下一個：是所有產業還是一個？**跨產業。**所以不是產業特定的。下一個：它們卡在審批流程的哪裡？**在支持者說了同意之後，但在採購批准之前。**」*

*「我打電話給 5 位交易停滯的支持者。他們都告訴我同一件事：採購需要一份 ROI 論證文件和一份安全問卷。我們的支持者沒有這些。他們卡在等我提供。我建立了一份標準化的 ROI 模板和預填的安全問卷。5 萬美元以上交易從提案到成交的平均時間從 45 天降到 22 天。」*

**人員管理者：「跟我說一個你診斷團隊表現不佳原因的經驗。」**

*「我的團隊的衝刺速度在三個衝刺週期中下降了 30%。我需要找出什麼改變了。」*

*「第一：是所有團隊成員還是特定的？**六人中有三人下降了。**另外三人穩定。下一個：那三人有什麼共同點？**是的。他們三人都在三個衝刺週期前開始的一個新的跨職能專案上。**下一個：那個專案是增加到他們的工作量還是取代現有工作？**增加。沒有人移除他們現有的衝刺承諾。**」*

*「根本原因是容量，不是績效。三位工程師被分配到一個副專案，卻沒有調整他們的衝刺負載。我和專案負責人合作，要麼減少他們的衝刺承諾，要麼為副專案引進一位約聘人員。速度在兩個衝刺週期內恢復到基準的 95%。」*

> **以我的經驗來說：** 我面試過最好的求職者都會做某種版本的這件事。他們不會說「問題是 X。」他們會帶我走過**他們是怎麼搞清楚**問題是 X 的。診斷過程比解決方案更令人印象深刻。

## 假設樹：不要急著下結論

**假設**是一種預感。它是你在確認之前認為答案是什麼。渡邊健介的重大教訓：**陳述你的假設，解釋你的推理，然後測試它。**

大多數人跳過測試的部分。他們把第一個直覺當成事實就跑了。

在書中，蘑菇戀人假設只有 **5%** 的學校知道他們的演唱會。真實數字是 **30%。**

他們假設知道的人中有 **60%** 會來。真實數字是 **10%。**如果他們在沒有測試的情況下就按照假設行動，他們會浪費時間在知名度宣傳上，而真正的問題完全是另一回事。

### 假設樹的結構

```
  假設樹
  ==========

           ┌─────────────────────────┐
           │      主要假設           │
           │                         │
           │  「我相信根本            │
           │   原因是 X。」          │
           └────────────┬────────────┘
                        │
          ┌─────────────┼─────────────┐
          │             │             │
   ┌──────┴──────┐ ┌───┴────┐ ┌──────┴──────┐
   │   原因 A    │ │原因 B  │ │   原因 C    │
   │             │ │        │ │             │
   │ 「因為數據  │ │「因為  │ │ 「因為類似  │
   │  顯示      │ │ 我之前 │ │  案例顯示   │
   │  這樣。」  │ │ 看過   │ │  這樣。」   │
   │             │ │ 這種   │ │             │
   │             │ │情況。」│ │             │
   └─────────────┘ └────────┘ └─────────────┘

  如果一塊磚很弱，整個金字塔就會搖晃。
  那是你的信號，要在行動前收集更多數據。
```

### 如何在面試中使用假設樹

當面試官問情境題（「如果……你會怎麼做？」），這樣做：

1. **陳述你的假設。** 「我的第一直覺是問題是 X。」
2. **解釋你的推理。** 「我這麼想是因為 A、B 和 C。」
3. **解釋你會如何測試。** 「在我投入那個方法之前，我會先確認 D 和 E。」

**範例：「如果一個產品上市表現不佳，你會怎麼做？」**

*「我最初的假設會是定位沒有和目標受眾產生共鳴。我會這麼想是因為流量數字強勁但轉換率低。人們有看到它但沒有購買。」*

*「但我不會馬上按照那個假設行動。我會透過檢視到達頁面的分析數據、確認漏斗中的流失點、以及和 5 到 10 位瀏覽過但沒有購買的客戶談話來測試。」*

*「如果數據確認了我的假設，我會重新設計訊息傳達。如果不是，我會把定價或產品市場適配度作為替代解釋來檢視。」*

**另一個範例：「如果員工敬業度分數下降，你會怎麼做？」**

*「我的假設是下降和最近的組織變動有關。我們兩個月前做了組織重整，以我的經驗，重整後敬業度會下降，因為人們對自己的角色感到不確定。」*

*「要測試這一點，我會按部門看數據。如果下降集中在被重整的團隊，我的假設成立。如果是全面性的，原因就是其他東西。我也會拉出問卷中的開放式回答，看看有什麼主題浮現。」*

*「如果數據確認是重整相關，我會建議做跨級會議，讓領導層直接處理疑慮。如果原因不同，我會回到數據，形成新的假設。」*

**另一個範例：「一個重要客戶威脅要離開。你的做法是什麼？」**

*「我的假設是客戶有一個未解決的問題，已經累積了一段時間。客戶很少因為單一事件就離開。通常是一個模式。」*

*「我會透過檢視過去 6 個月的溝通記錄、客服工單和會議紀錄來測試。我也會直接打電話給客戶，問一個開放式問題：『需要改變什麼才能讓你有信心繼續和我們合作？』」*

*「他們的回答要麼確認我的假設，要麼給我新的資訊。無論哪種方式，我的下一步都是基於真實數據，而不是猜測。」*

### 什麼時候使用假設樹

- 「如果……你會怎麼做？」
- 「你會如何處理這個情境？」
- 「你覺得是什麼造成了這個問題？」
- **任何要求你推測、評估或建議的問題。**

### 各職位的假設樹

**工程：「如果你的微服務開始拋出間歇性錯誤，你會怎麼做？」**

*「我的假設是這是資源競爭問題，而不是程式碼 bug。在本地無法重現的間歇性錯誤通常和基礎設施有關：記憶體限制、連線池耗盡、或在負載下的競爭條件。」*

*「我會透過檢查三件事來測試。第一，錯誤發生期間服務的記憶體和 CPU 指標。第二，資料庫連線池使用率。第三，錯誤是否和流量高峰相關。如果錯誤在尖峰流量時發生且連線池達到上限，我的假設成立，修復方法是增加連線池大小或加入斷路器。如果錯誤和負載無關是隨機的，我會把假設轉向程式碼層級的競爭條件，檢視最近的提交。」*

**營運：「如果一個關鍵供應商錯過了交貨期限，你會怎麼做？」**

*「我的假設是這是他們那邊的產能問題，而不是一次性的失誤。錯過期限的供應商通常是在發出信號，需求已經超過了他們的生產能力。」*

*「我會透過問兩個問題來測試。第一，他們是否也對其他客戶錯過了期限？如果是，這是系統性的產能問題。第二，我們最近的訂單量是否增加了？如果我們成長了 30% 但沒有和他們談過擴展，我們也是問題的一部分。」*

*「短期，我會和他們談判一個有具體日期的恢復計畫。長期，我會評估是否需要第二個供應商來降低單一來源風險。我也會在這個供應商的零件庫存模型中加入 2 週的緩衝。」*

**客戶服務：「如果來電量突然翻倍，你會怎麼做？」**

*「我的假設是有什麼特定的事情觸發了它。來電量不會隨機翻倍。通常是產品變更、服務中斷、帳單週期或外部事件驅動的。」*

*「我會透過查看過去 48 小時發生了什麼來測試。我們推了產品更新嗎？帳單週期到了嗎？有關於 bug 的社群媒體貼文瘋傳嗎？我也會拉出過去 24 小時的來電原因代碼，和上一週比較。如果 60% 的新來電是關於同一個主題，那就是觸發因素。」*

*「一旦找到觸發因素，回應方式會改變。如果是 bug，我會升級到工程團隊並在 IVR 上放一條訊息讓客戶知道我們已經知道了。如果是帳單週期，我會安排接下來 3 天的加班人力。用同一種方式處理所有來電高峰是個錯誤。」*

**業務：「如果你的勝率突然下降，你會怎麼做？」**

*「我的假設是競爭環境或我們自己的銷售流程中有什麼改變了，而不是團隊一夜之間變差了。勝率不會因為努力不夠而下降 10 個百分點。」*

*「我會透過查看失敗交易數據來測試。我們是否比以前更常輸給某個特定競爭對手？如果是，他們改了定價或推出了新功能。下降是發生在所有業務代表還是部分？如果是所有人，原因是外部的。如果是特定的人，原因是內部的。」*

*「我也會聽 10 通最近的失敗交易通話錄音。找出你為什麼在輸的最快方式，是聽潛在客戶選擇別人時說了什麼。」*

**人員管理者：「如果多個團隊成員在同一個月遞出辭呈，你會怎麼做？」**

*「我的假設是有一個共同的根本原因。當多人同時離開，這很少是巧合。通常和最近的變動有關：新的領導、組織重整、薪酬問題或一個被打破的承諾。」*

*「我會透過做兩件事來測試。第一，我會和每個人進行誠實的離職對話。不是 HR 的離職面談。是真正的對話，我會問：『壓垮駱駝的最後一根稻草是什麼？』第二，我會看 2 到 3 個月前有什麼改變，因為辭職決定通常會在觸發事件之後延遲幾個月。」*

*「如果根本原因是我能解決的，我會立即且透明地和剩餘團隊處理。如果是我無法控制的，我會專注在和我最高風險的剩餘團隊成員進行留任對話，在他們也開始找之前。」*

> **以我的經驗來說：** 當求職者說「我認為問題是 X，以下是我會如何驗證」，我知道他們在工作中不會做衝動的決定。那一句話傳達出**成熟、謙遜和強大的判斷力。**這很少見。用起來吧。

## 正確的優缺點分析：如何回答「為什麼選我們？」和「為什麼選這個職位？」

在書中，足球少女奇異果要在巴西兩所學校之間做選擇。她差點選錯了。為什麼？她用了**錯誤的標準**來比較。她過度重視不重要的事（地點、網站設計），卻低估了重要的事（教練品質、上場時間、文化沉浸）。

一位體育評論員的分析讓她重新思考一切：**「強大的球隊不一定等於好的成長環境。」**她回去用正確的標準重新建立評估，並正確加權。

```
  奇異果的錯誤 vs. 奇異果的修正
  ===============================

  第一次評估（錯誤標準）：
  ┌────────────────────┬──────────┬──────────┐
  │ 標準               │ 里約中學 │ 亞馬遜中學│
  ├────────────────────┼──────────┼──────────┤
  │ 好看的網站         │ +++++    │ +        │
  │ 在大城市           │ +++++    │ +        │
  │ 較便宜的學費       │ +++      │ +        │
  │ 贏家：里約         │          │          │
  └────────────────────┴──────────┴──────────┘

  第二次評估（正確標準，加權）：
  ┌────────────────────┬──────────┬──────────┬──────────┐
  │ 標準               │ 權重     │ 里約中學 │ 亞馬遜中學│
  ├────────────────────┼──────────┼──────────┼──────────┤
  │ 上場時間           │ 高       │ ++       │ +++++    │
  │ 教練品質           │ 高       │ ++       │ +++++    │
  │ 文化沉浸           │ 高       │ ++       │ +++++    │
  │ 球隊實力           │ 高       │ +++++    │ +++++    │
  │ 接近里約           │ 低       │ +++++    │ +        │
  │ 贏家：亞馬遜       │          │          │          │
  └────────────────────┴──────────┴──────────┴──────────┘
```

面試中也會發生同樣的錯誤。求職者說「我想在你們公司工作，因為你們是很好的品牌」或「我很喜歡你們的使命。」這些是**表面回答。**它們就相當於奇異果因為網站好看就選了里約中學。

### 渡邊健介的標準與評估方法

**步驟一：** 列出你的選項。
**步驟二：** 定義你的評估標準。
**步驟三：** 按重要性為每個標準加權（高、中、低）。
**步驟四：** 根據加權標準為每個選項評分。
**步驟五：** 選擇最具吸引力的選項。

### 如何應用在「為什麼選我們？」的問題上

**範例：「你為什麼想在這裡工作？」**

**弱回答：** *「我很喜歡你們的品牌。使命和我產生共鳴，我覺得文化很棒。」*

**強回答：** *「我根據目前對我最重要的三個標準來評估這個職位。第一，**在規模化問題上工作的機會。**你們的團隊服務 5000 萬月活用戶。那是我成長最快的環境。第二，**團隊結構。**從面試過程中了解到，你們的團隊讓成員負責端到端的專案，而不是把工作切成狹窄的任務。第三，**成長軌跡。**這個職位直接對應到我 3 年後想到達的地方。」*

*「我把這些標準的權重設得很高，因為在我職業生涯的這個階段，成長和自主權對我來說比品牌名稱或薪酬更重要。」*

**範例：「你為什麼選我們而不是 X 公司？」**

*「我根據三個標準評估了兩個機會。第一，職位範圍。你們的職位負責從獲客到留存的完整漏斗。另一個職位只專注在漏斗頂端。第二，我會從中學習的團隊。我和你們團隊的兩個人聊過，他們在數據驅動成長方面的專業深度讓我印象深刻。第三，交付速度。你們每週部署。另一家公司按季度交付。以我目前的職涯階段，我需要快速的回饋循環。」*

**範例：「你為什麼選擇你目前的方法而不是其他替代方案？」**

*「我有兩個選項。選項 A 是從頭重建系統。選項 B 是修補現有系統。我根據三個標準評估：實施時間、破壞現有功能的風險、以及長期可維護性。選項 A 在可維護性上得分最高，但在時間和風險上最低。選項 B 正好相反。考慮到我們有一個 4 週的硬性截止日，我選了選項 B。但我記錄了技術債，並安排了第二季的重建。」*

### 奇異果的教訓：挑戰你自己的思考

奇異果的故事教了另一個重要的面試教訓：**挑戰你自己的結論。**

當你準備面試時，問自己：
- 我選擇這家公司的理由對嗎？
- 我是否過度重視表面因素（品牌、辦公室、福利）？
- 我是否低估了成長因素（團隊品質、主管、學習機會）？
- 我是否收集了足夠的資訊來做出好的決定？

奇異果聯繫了兩所學校的在校生。她打電話給教練。她收集了真實數據，而不是依賴網站上看起來好的東西。**照做吧。**和在那家公司工作的人聊聊。在面試中問真正的問題。用數據做決定，不要用假設。

> **以我的經驗來說：** 我總是能分辨求職者做了深入研究還是表面研究。表面研究者說「我很喜歡你們的文化。」深入研究者說「我和你們團隊的兩個人聊過，他們都提到了每週回饋會議。那正是我做出最佳表現的環境。」其中一個答案會拿到錄取。**猜猜是哪一個。**

## 蘑菇戀人方法：在面試和案例研究報告中使用數據

這一節是書中框架**最直接有用**的地方。蘑菇戀人的演唱會出席故事是一個完整的案例研究：問題識別、假設建立、數據收集、分析和解決方案開發。

**如果你在面試中需要做案例研究報告，這就是你的藍圖。**

### 蘑菇戀人流程（回顧）

```
  蘑菇戀人：從 15 人到 200 名觀眾

  ┌──────────────────────────────────────────────────────┐

  │ 「只有 15 個人來看我們的演唱會。」                   │

  ├──────────────────────────────────────────────────────┤
  │ 步驟三：收集數據                                     │
  │ 在早自習時進行的三題問卷調查。                       │
  ├──────────────────────────────────────────────────────┤
  │ 步驟四：將假設與現實比較                             │

  │ 總出席人數：15 --> 200                               │
  │ 回頭率：80% --> 90%                                  │
  └──────────────────────────────────────────────────────┘

### 如何應用在案例研究面試

很多公司會要求求職者準備案例研究報告。他們給你一個商業問題，要求你報告你的分析和建議。

以下是從蘑菇戀人方法提取的精確結構。**每個部分我都附上了完整的範例回答。**

#### 第一部分：清楚定義問題

不要從解決方案開始。從問題開始。用**具體、可衡量的術語**陳述。

**差：** 「公司需要更多客戶。」
**好：** 「過去兩個季度，每月新客戶獲取量下降了 25%，從每月 800 人降到 600 人。」

**範例陳述：** *「我們要解決的問題是：過去兩個季度，每月新客戶註冊量從 800 人下降到 600 人。這是 25% 的下降，相當於大約 12 萬美元的每月經常性收入損失。」*

#### 第二部分：列出所有可能的原因

使用**分解樹。**把問題分成分支。展示你已經考慮了所有角度。

**範例：** *「客戶獲取是一個漏斗。我把下降分成四個階段：**知名度**（人們看到我們了嗎？）、**考慮**（他們有興趣嗎？）、**轉換**（他們在購買嗎？）、和**留存**（他們買了之後還留著嗎？）。下降發生在任何一個階段都有可能。」*

#### 第三部分：陳述你的假設

清楚說明你認為問題是什麼以及為什麼。使用渡邊健介的**假設樹。**

**範例：** *「我的假設是下降發生在轉換階段。流量穩定在每月 5 萬訪客。廣告支出沒有改變。但轉換率從 4% 下降到 2.5%。這告訴我人們有找到我們但沒有購買。」*

#### 第四部分：展示你的數據收集和分析

解釋你收集了什麼數據或會收集什麼數據。解釋它顯示了什麼。

**範例：** *「我按渠道、到達頁面、裝置類型和客戶群來分析轉換率。數據顯示轉換率**只在行動端下降了。**桌面轉換率維持在 4.2%。行動端轉換率從 3.8% 降到 1.2%。由於行動端占我們 60% 的流量，這一個變化解釋了整個下降。」*

#### 第五部分：將假設與現實比較

展示你測試了你的假設。**這是區分好的分析師和優秀分析師的地方。**

**範例：** *「我最初的假設部分正確。轉換確實是問題，但根本原因比我預期的更具體。不是訊息傳達或定價的問題。當我自己測試行動端結帳流程時，我發現了一個 bug：在上次應用程式更新後，iOS 裝置上的『完成購買』按鈕無法載入。」*

#### 第六部分：推薦解決方案並排優先順序

列出你的想法。排優先順序。使用書中的**影響力 vs. 執行難度矩陣。**

```
  影響力 vs. 執行難度矩陣
  ========================

  高影響力  │ 修復行動端        │ A/B 測試新的
            │ 結帳 bug          │ 行動端結帳流程
            │ [立即做]          │ [下一季做]
  ──────────┼───────────────────┼───────────────────
  低影響力  │ 重新設計整個      │ 在發布流程中
            │ 行動應用程式      │ 加入 QA 步驟
            │ [暫時不做]        │ [這個月做]
  ──────────┼───────────────────┼───────────────────
            │ 困難              │ 容易
            │     執行難度
```

**範例：** *「我有三個建議。第一，**修復行動端結帳 bug。**高影響力，容易實施。今天就做。這一項應該能恢復大部分失去的轉換。第二，**在發布流程中加入行動端 QA 檢查清單。**中低影響力但容易做。防止再次發生。第三，**A/B 測試簡化的行動端結帳流程。**高影響力，中等工作量。下一季做，讓行動端轉換率超越之前的基準線。」*

#### 第七部分：展示預期結果

用預測結果收尾。使用真實或估計的數字。

**範例：** *「如果我們修復行動端結帳並恢復行動端 3.8% 的轉換率，我們大約能恢復**每月 150 位客戶。**那是每月 9 萬美元的經常性收入。加上 QA 流程改變，我們估計能降低 80% 再次發生的風險。」*

### 蘑菇戀人原則適用於所有面試回答

即使你不是在做正式的案例研究，這個原則適用於每一個行為面試回答：**帶上數據。**

## 章魚約翰方法：在面試中展示目標設定

面試官喜歡問**「你五年後看到自己在哪裡？」**和**「你的職涯目標是什麼？」**

大多數求職者給出夢想家式的回答。「我想在領導職位上。」「我想要創造影響力。」這些答案聽起來不錯，但什麼都沒說。

在書中，章魚約翰想買一台電腦。他第一個目標陳述很弱：「我想要一台電腦。」渡邊健介解釋為什麼那不是一個真正的目標。**一個真正的目標要回答三個問題：**

```
  章魚約翰的目標設定方法
  =======================

  差的目標：「我想要一台電腦。」

  好的目標：「我想要在六個月內，不向別人借錢
              的條件下，買一台 500 美元的二手
              Apple 電腦。」

  ┌──────────────────────────────────────────┐
  │  1. 具體要什麼？    500 美元二手 Apple   │
  │  2. 什麼時候？      六個月內              │
  │  3. 什麼條件？      不借錢                │
  └──────────────────────────────────────────┘

  然後：計算差距

  ┌────────────────────────────────────────────┐
  │  目標：              $500                   │
  │  預計存款：          $252                   │
  │  差距：              $248                   │
  │                                            │
  │  填補差距的計畫：                          │
  │  + 減少支出：        +$60                  │
  │  + 賣掉不用的東西：  +$50                  │
  │  + 同時遛三隻狗：    +$138                 │
  │  = 差距填補完成                            │
  └────────────────────────────────────────────┘
```

### 如何回答「你五年後看到自己在哪裡？」

使用約翰的方法。對**什麼、什麼時候、和怎麼做**要具體。

**弱回答：** *「我看到自己在一個領導角色。」*

**強回答：** *「在接下來的 2 年，我想成為團隊中數據分析的核心人物。我想端到端地負責至少一個主要專案。到了第 3 年，我想開始指導較新的團隊成員。到第 5 年，我想帶領一個小團隊或負責一個主要工作流。」*

*「我目前和目標之間的差距主要在兩個方面：**專案領導經驗**和**人員管理技能。**這個職位填補了第一個差距。我計畫透過志願參加導師計畫和跨職能專案來填補第二個。」*

**另一個範例：「你的職涯目標是什麼？」**

*「我的 2 年目標是成為一個能端到端負責產品領域的資深產品經理。目前我是一個執行力強但策略方面還在發展中的 PM。差距是損益表管理和更大規模的跨職能領導經驗。你們公司的這個職位填補這個差距，因為這裡的 PM 負責完整的產品生命週期，並直接和工程、設計和行銷合作。」*

### 用差距分析來進行職涯對話

約翰的差距分析也適用於**「你為什麼想換工作？」**這個問題。

不要抱怨你現在的工作（批評小子），把它框架成一個差距：

**弱回答（抱怨者）：** *「我現在的公司毫無前途。沒有成長空間。我的主管不支持我。」*

**強回答（差距分析）：** *「我的目標是在全球規模的問題上工作。我目前的角色專注在單一市場。我和目標之間的差距是**國際經驗和跨市場策略。**你們的團隊在 15 個國家運作。那正是我需要填補的差距。」*

**另一個範例：**

*「我想從專案管理轉到產品管理。差距在於產品相關技能：客戶研究、路線圖優先排序、和與工程團隊合作制定功能規格。我一直在用業餘時間上 PM 課程和經營副業專案來填補這個差距。但最快填補差距的方式是全職做 PM。這就是我來這裡的原因。」*

沒有負面情緒。沒有抱怨。**純粹的問題解決邏輯。**面試官喜歡這個框架。

> **以我的經驗來說：** 當求職者把職涯轉換解釋為填補一個具體差距時，它告訴我兩件事。第一，他們對自己的職涯是有意圖的。第二，他們不會一時衝動就離開這個職位。他們是有原因來的。這兩個信號都讓我更有信心推薦他們。

## 問題解決者永遠不會犯的 5 個面試錯誤

這五個錯誤都來自書中的核心教訓。每一個都對應到渡邊健介警告的思考錯誤。

```
  五個錯誤和修正
  ===============

  錯誤                          修正
  ───────────────────────────  ───────────────────────────
  1. 在理解問題之前             暫停。釐清。重述。
     就回答                     然後回答。

  2. 解決表面症狀，             持續問「為什麼？」
     而不是根本原因             直到找到真正的原因。

  3. 把第一個直覺               陳述你的假設。
     當成事實                   然後解釋你如何測試它。

  4. 給一個解決方案             展示 2-3 個選項。
     但面試官想看你的           解釋為什麼選了其中一個。
     思考過程

  5. 做決定時沒有               說出你的標準。
     明確的標準                 解釋權重。
```

### 錯誤一：在理解問題之前就回答

這是跳過框架的**步驟一。**你聽到問題就開始說話，卻不知道面試官在問什麼。

**錯誤做法：** *「跟我說一個你領導的專案。」求職者馬上開始講故事，沒有問面試官關心哪種類型的專案。*

**正確做法：** *「你說的『專案』，你是在找一個技術專案、跨職能計畫、還是我領導人員的經驗？三種我都有例子，我想給你最相關的那個。」*

### 錯誤二：治療症狀而不是根本原因

書中那個數學成績下降的學生考慮退出足球隊來增加學習時間。但根本原因不是學習時間。是**特定的幾何題型。**解決方案是有針對性的練習，不是放棄足球。

**錯誤做法：** *「我們的團隊很慢，所以我推動每個人更努力工作。」*

**正確做法：** *「我們的團隊很慢。我深入數據，發現 70% 的延遲來自一個瓶頸：等待設計審查。設計師們不是慢。他們是超載了，因為他們同時支援三個團隊。我提議為我們的團隊配置一位專屬設計師。交付時間下降了 60%。」*

### 錯誤三：沒有測試你的假設

蘑菇戀人的假設是錯的。他們以為知名度是 **5%。**是 **30%。**如果他們不測試就行動，他們會浪費時間。

**錯誤做法：** *「客戶不開心，所以我重新設計了產品。」*

**正確做法：** *「我有個預感客戶對產品不滿意。在重新設計任何東西之前，我拉了 NPS 數據，檢視了客服工單，並訪談了 10 位客戶。結果他們很喜歡產品。他們不滿的是帳務體驗。修復比全面重新設計小得多。」*

### 錯誤四：面試官想看你的思考過程時只給一個方案

分解樹能產生很多可能的方案。蘑菇戀人列了 **13 個想法。**然後排了優先順序。

**錯誤做法：** *「所以我把報表自動化了。」*

**正確做法：** *「我考慮了三個選項。選項 A：完全自動化報表。選項 B：簡化報表模板，讓手動填寫更快。選項 C：取消報表，用儀表板取代。我選了選項 C，因為這份報表只有兩個人在看，他們告訴我他們想要即時數據，不是每週的 PDF。」*

### 錯誤五：做決定時沒有加權標準

奇異果差點選錯學校，因為她用了錯誤的標準。

**錯誤做法：** *「感覺是對的選擇。」*

**正確做法：** *「我根據三個標準評估了兩家供應商：整合速度、持續成本和客服回應速度。供應商 A 比較便宜，但需要 6 週的整合時間。供應商 B 貴了 20%，但一週內就能整合，而且有全天候客服。考慮到我們緊迫的截止日，我把速度和支援的權重設得比成本高，選了供應商 B。」*

## 框架實戰：各職位的範例

這份指南中的每一個工具都適用於每一種職位。但語言會因為你面試的是工程、營運、客服、業務還是管理職位而改變。

以下是同一個框架在五個不同職位中的表現。每個範例都使用完整的四步驟流程。**我標註了回答的每一個部分，讓你看到哪個步驟在發揮作用。**

### 工程

**問題：「跟我說一個你解決複雜技術問題的經驗。」**

**[步驟一 - 了解]** *「我們的 API 回應時間在兩週內從 200 毫秒跳到 1.2 秒。團隊表面上的假設是我們需要擴展伺服器。」*

**[步驟二 - 根本原因]** *「在那樣做之前，我想了解根本原因。我把問題拆成三個領域：資料庫查詢、應用程式邏輯和網路延遲。我在每一層做了追蹤。資料庫查詢時間持平。網路延遲持平。整個飆升完全在應用程式層。」*

**[步驟二 - 根本原因（深入）]** *「我進一步縮小範圍。我看了哪些端點最慢。一個端點占了 85% 的惡化。那個端點在兩週前被修改過，當時我們加了一個新功能。新代碼對列表中的每個項目都呼叫了一個外部服務的巢狀迴圈。對小列表來說沒問題。對超過 50 個項目的列表，它會累積成數秒的等待時間。」*

**[步驟三 - 行動計畫]** *「我有兩個選項。選項 A：為外部服務呼叫加入快取。實施快速但會提供最多 5 分鐘的過期數據。選項 B：重構代碼，把外部呼叫批量處理成單一請求。更乾淨的方案但代碼改動更大。我選了選項 B，因為這個功能的數據準確性很重要。我也在 CI 管線中加了效能測試，這樣這類退化在部署前就能被捕捉到。」*

**[步驟四 - 執行和結果]** *「結果：回應時間降回 180 毫秒，比原始基準線還好。新的 CI 效能測試之後已經標記了三次潛在的退化，全部在進入正式環境前被攔截。」*

```
  工程回答對應框架
  =================
  步驟一：「API 回應時間從 200ms 跳到 1.2s。」
  步驟二：「拆成 3 層。飆升在應用程式層。
           一個端點，一個巢狀迴圈，一個新功能。」
  步驟三：「兩個選項。快取 vs. 批量處理。
           因為數據準確性選了批量處理。」
  步驟四：「回應時間：1.2s --> 180ms。
           加了 CI 測試。之後攔截了 3 次退化。」
```

**問題：「你會如何設計一個處理 10 倍當前流量的系統？」**

*「我會把這拆成三個領域。第一，**找到瓶頸。**系統在 10 倍負載下最先在哪裡崩潰？是資料庫、應用伺服器還是網路？我會在設計任何東西之前先跑負載測試。」*

*「第二，**評估瓶頸的選項。**如果是資料庫，我會看讀取副本、快取層或分片。如果是應用程式層，我會看水平擴展、非同步處理或優化熱路徑。如果是網路，我會看 CDN 或邊緣運算。」*

*「第三，**按影響力和風險排優先順序。**加一個快取層是高影響力低風險。分片資料庫是高影響力但高風險和高複雜度。我會從影響力最高、風險最低的改變開始，衡量後再做下一個。」*

*「我對大多數 10 倍規模的網路應用程式的假設是，資料庫是第一個瓶頸。我會先用負載測試驗證，然後再投入任何工程時間。」*

### 營運

**問題：「跟我說一個你改善營運流程的經驗。」**

**[步驟一 - 了解]** *「我們暢銷 SKU 的缺貨率在第二季達到 18%。這意味著近五分之一的最佳產品在客戶要購買時無法取得。公司目標是低於 5%。我的主管要求我解決它。」*

**[步驟二 - 根本原因]** *「我拉了 90 天的庫存數據，把缺貨分成三個類別：預測錯誤（我們訂太少）、供應商延遲（我們按時下單但供應商交貨延遲）、和需求暴增（不可預測的激增消耗了安全庫存）。」*

**[步驟二 - 根本原因（深入）]** *「預測錯誤占了缺貨的 60%。供應商延遲是 25%。需求暴增只有 15%。所以最大的問題是我們自己的預測，不是供應商。我進一步深入。我們的預測模型使用簡單的 3 個月滾動平均。它沒有考慮季節性、促銷活動或趨勢變化。每次行銷團隊跑活動，我們的庫存消耗速度都比模型預測的快。」*

```
  營運：什麼造成了缺貨？
  ========================

  預測錯誤     ██████████████████████████████  60%
  供應商延遲   █████████████████  25%
  需求暴增     ██████████  15%

  ^^^^ 根本原因：我們的預測模型沒有考慮
       促銷活動或季節性。
```

**[步驟三 - 行動計畫]** *「我提出了三個選項。選項 A：聘請一位需求規劃師。高影響力但需要 3 個月的招聘時間和 9 萬美元薪資。選項 B：升級我們的預測模型，納入促銷行事曆和季節性調整。我有 Excel 技能，兩週內做出 v1。選項 C：所有 SKU 的安全庫存增加 30%。容易實施但佔用 20 萬美元的營運資金。」*

*「我建議選項 B 作為主要修復，同時在新模型建置期間對前 20 個 SKU 暫時增加安全庫存。選項 A 作為長期方案。」*

**[步驟四 - 執行和結果]** *「我在兩週內建好了新的預測模型。它拉入了行銷團隊的促銷行事曆，並根據前一年的數據套用季節性乘數。缺貨率在第一個月從 18% 降到 6%。到第三季末我們達到 3.8%，低於 5% 的目標。我們也在模型證明可靠後釋放了額外的安全庫存，收回了 15 萬美元的營運資金。」*

```
  營運回答對應框架
  =================
  步驟一：「缺貨率 18%。目標低於 5%。」
  步驟二：「60% 由預測錯誤造成。模型使用
           簡單平均。沒有考慮促銷或季節性。」
  步驟三：「3 個選項：聘人、升級模型、增加庫存。
           選了模型升級（2 週）+ 對前 20 個 SKU
           暫時增加安全庫存。」
  步驟四：「缺貨率：18% --> 6%（一個月）--> 3.8%
           （第三季末）。收回 15 萬美元營運資金。」
```

**問題：「如何處理你目前營運無法支撐的突發需求暴增？」**

*「在反應之前，我會問三個問題。第一，這是一次性的暴增還是趨勢？那決定了我需要臨時解方還是永久方案。第二，限制在哪裡？是人、空間、設備還是供應商？第三，無法滿足需求的代價是什麼？我們是永遠失去客戶還是暫時延遲訂單？」*

*「我對大多數需求暴增的假設是，限制在營運的某一個特定環節，而不是所有地方。我會先找到那個瓶頸。然後看臨時解方：加班、臨時工、暴增期間簡化流程、或合作夥伴履行。我會按實施速度排優先順序，因為需求暴增是時間敏感的。」*

*「長期來看，我會把這次暴增的數據納入我們的產能規劃模型，這樣下次就有劇本。」*

### 客戶服務

**問題：「跟我說一個你扭轉負面客戶體驗的經驗。」**

**[步驟一 - 了解]** *「一位客戶打進來非常憤怒，因為他們在 3 天內被轉接了 4 次，試圖解決一個帳務錯誤。他們被同一筆訂單收了兩次費。到他們找到我的時候，已經準備要取消帳戶了。」*

**[步驟二 - 根本原因]** *「在跳到解決方案之前，我需要了解全貌。我拉出了他們的帳戶歷史，確認了雙重收費。然後我查看了之前的通話紀錄，了解為什麼之前沒有解決。原來前兩位客服無法在沒有主管批准的情況下處理超過 100 美元的退款。第三位客服提交了批准請求，但它卡在排隊中。沒有人把狀態回報給客戶。」*

**[步驟二 - 根本原因（深入）]** *「根本原因不是帳務錯誤。帳務錯誤會發生。根本原因是交接流程。沒有人負責到底。每個客服都把它當成一張新工單，而不是跟進現有的那張。」*

**[步驟三 - 行動計畫]** *「我採取了三個行動。第一，我當場處理退款。我在通話中打給主管，即時取得批准，而不是等排隊。第二，我給客戶 50 美元的信用額度作為補償，以及我的個人電子郵件，以防他們未來有問題。第三，我寫了一份案例報告，分享給我的組長，作為為什麼我們的交接流程需要改進的範例。」*

**[步驟四 - 執行和結果]** *「客戶留下來了。他們留了正面評價，提到了我的名字。我的組長用我的報告建立了一個新流程：任何未解決的帳務問題都會指派給一個單一負責人，跟進到完成為止。帳務類的重複升級在下一季下降了 40%。」*

```
  客服回答對應框架
  =================
  步驟一：「客戶在 3 天內被轉接 4 次。
           雙重收費。準備取消。」
  步驟二：「客服無法處理超過 100 美元的退款。
           批准卡在排隊中。沒有人溝通。
           根本原因：交接流程，不是帳務錯誤。」
  步驟三：「3 個行動。立即退款。信用額度 + 直接郵件。
           撰寫案例報告推動流程改變。」
  步驟四：「客戶留下來了。留了正面評價。
           建立了新的單一負責人流程。
           帳務升級下降 40%。」
```

**問題：「你會如何在不影響客戶滿意度的情況下降低平均處理時間？」**

*「我會從了解時間花在哪裡開始。我會抽取 50 通電話的樣本，把處理時間拆成幾個類別：了解問題、系統查詢、實際解決和結案。」*

```

  了解問題          ████████  2.5 分鐘
  系統查詢          ████████████████  5.0 分鐘
  實際解決          ██████  2.0 分鐘

                    ─────────────────────
  平均總計：        12 分鐘

  最大機會：系統查詢（占總時間 42%）
```

*「我的假設是系統查詢是最大的時間消耗。如果客服花 5 分鐘在系統之間切換來找客戶資訊，那是工具問題，不是人的問題。告訴客服『講快一點』不會解決這個問題。」*

*「我會提議一個統一的客戶儀表板，把帳戶、訂單和帳務數據拉到一個螢幕。這能把查詢時間減半。我也會看前 10 大來電原因，為最常見的原因建立一鍵解決路徑。這兩個改變應該能降低處理時間，同時改善體驗，因為客戶也能更快得到答案。」*

### 業務

**問題：「跟我說一個你扭轉表現不佳的區域或銷售管道的經驗。」**

**[步驟一 - 了解]** *「我接手了一個連續 3 個季度未達標的區域。前任業務代表被解僱了。我的主管的指示是：『快點修好。』」*

**[步驟一 - 了解（範圍界定）]** *「在我開始打電話之前，我需要了解為什麼表現不佳。我把銷售管道按階段拆開，和公司基準比較。」*

```
  業務：銷售管道診斷
  ====================

  階段            我的區域        公司平均
  ─────────────  ────────────    ───────────────
  每月潛在客戶    120             100
  約到會議        15 (13%)        20 (20%)
  發出提案        8 (53%)         12 (60%)
  成交            2 (25%)         5 (42%)
  ─────────────  ────────────    ───────────────

  發現：潛在客戶量沒問題。每個階段的轉換率
  都低於平均。最大差距：成交率。
```

**[步驟二 - 根本原因]** *「潛在客戶量高於平均。問題不在開發。每個階段的轉換率都低於公司平均，最大的差距在成交率：25% 對比 42% 的公司基準。」*

**[步驟二 - 根本原因（深入）]** *「我進一步深入。我檢視了 10 筆最近失去的交易。10 筆中有 7 筆有同樣的模式：潛在客戶在收到提案後就消失了。我打電話給 5 位失去的潛在客戶，問他們發生了什麼。三位說提案太通用了。兩位說競爭對手提供了更客製化的方案。」*

**[步驟二 - 根本原因（確認）]** *「根本原因不是努力不夠或量不夠。是提案品質。前任業務代表發同一份範本給每個潛在客戶，沒有針對他們的具體痛點做客製化。」*

**[步驟三 - 行動計畫]** *「我重建了提案流程。我建立了一個提案前的需求確認電話，確認前 3 大痛點和決策標準。然後我針對那些具體的點客製化每一份提案。我也加入了和潛在客戶的產業匹配的客戶案例。」*

**[步驟四 - 執行和結果]** *「一個季度內，我的成交率從 25% 提升到 48%。那一季我達成了 110% 的配額，之後一整年都沒有再落後。區域的總營收從年 80 萬美元增長到 140 萬美元。」*

```
  業務回答對應框架
  =================
  步驟一：「區域連續 3 季未達標。
           把銷售管道按階段拆開 vs. 基準。」
  步驟二：「潛在客戶量沒問題。成交率 25% vs. 42%。
           打給 5 位失去的潛在客戶。提案太通用。
           根本原因：提案品質，不是努力。」
  步驟三：「重建提案流程。提案前需求確認電話。
           客製化提案。產業案例。」
  步驟四：「成交率：25% --> 48%。達成 110% 配額。
           營收：80 萬 --> 140 萬美元/年。」
```

**問題：「你會如何進入一個沒有現有客戶的全新市場？」**

*「我會分成三個階段。第一，**研究和假設。**這個市場的理想客戶是誰？我們為他們解決什麼問題？我會在嘗試銷售任何東西之前，先和 15 到 20 位潛在客戶交談。我會測試我對他們痛點的假設。」*

*「第二，**小規模測試。**我會找出 5 個符合理想輪廓的潛在客戶，和每一個做完整的銷售週期。目標不是成交全部 5 個。是學習在這個新市場中銷售流程在哪裡斷裂。我們的訊息有打中嗎？定價對嗎？會出現什麼反對意見？」*

*「第三，**擴展有效的方法。**從那 5 次對話中汲取教訓，改進推銷話術，建立一個可重複的手冊。然後增加外展量。」*

*「書中的蘑菇戀人做了完全一樣的事。他們沒有馬上對 500 個人宣傳演唱會。他們做了問卷，找出真正阻止人們來的原因，解決了那些問題，然後再擴展。這就是我進入新市場的方式。」*

### 人員管理者

**問題：「跟我說一個你處理表現不佳團隊成員的經驗。」**

**[步驟一 - 了解]** *「我的一位直接下屬，一位中階分析師，連續三個專案都錯過了截止日。最簡單的假設是他們不夠努力。我的跨級主管建議把他們列入績效改善計畫。」*

**[步驟二 - 根本原因]** *「在那樣做之前，我想了解根本原因。我用了是非樹的方法來診斷。」*

```

**[步驟二 - 根本原因（確認）]** *「我進行了一對一，問了開放式問題。『帶我走過你這週怎麼分配時間的。』結果發現，兩個月前一位隊友離開時，這個人吸收了所有的專案，但他們自己的工作沒有被重新分配。他們在做 1.5 個人的工作。」*

**[步驟三 - 行動計畫]** *「根本原因不是績效。是容量。我有三個選項。選項 A：不管怎樣把他們列入績效改善計畫，希望他們更努力工作。那會是衝動小子的做法，而且大概會導致他們辭職。選項 B：把額外的專案在團隊中重新分配。選項 C：在我們找到替補人選之前，引進一位約聘人員來補這個缺。」*

**[步驟四 - 執行和結果]** *「我選了 B 和 C 的組合。我立即把兩個低優先順序的專案重新分配給其他團隊成員。我也取得了一位 3 個月約聘人員的批准。兩週內，那位分析師回到了正軌。他們在那個季度剩餘時間內每個截止日都達成了，並在下一次一對一中告訴我他們之前差點辭職。」*

**[步驟四 - 教訓]** *「如果我跳過診斷就直接列入績效改善計畫，我會因為一個工作量問題而失去一位強大的員工。」*

```
  人員管理者回答對應框架
  ========================

  步驟二：「用是非樹診斷：技能？否。清晰度？
           否。容量？是。吸收了離職隊友的
           全部工作量。做 1.5 倍的工作。」
  步驟三：「3 個選項。PIP（差）、重新分配（好）、
           約聘人員（好）。選了 B + C 組合。」
  步驟四：「兩週內回到正軌。之後每個截止日都達成。
           留住了一位強大的員工。避免因為工作量
           問題失去人才。」
```

**問題：「你會如何從零開始建立一個團隊？」**

*「我會使用章魚約翰同樣的目標設定方法。從具體目標開始，找出差距，然後制定計畫。」*

*「**具體要什麼？**我需要一個 6 人團隊，能完整負責客戶成功功能：到職、採用、續約和擴展。**什麼時候？**4 個月內完整到位。**條件？**在批准的人力預算 65 萬美元總薪酬內。」*

*「然後我會計算差距。今天我有零個人。我需要 6 個。但不是所有 6 個人的優先順序都一樣。我會使用一個優先排序矩陣。」*

```
  人員管理者：招聘優先排序
  ==========================

  高影響力  | 資深客戶成功     | 客戶成功營運
            | 經理（負責續約   | （建立劇本、
            | 和培訓團隊）     | 報表）
            | [第 1 個月聘]    | [第 3 個月聘]
  ──────────|──────────────────|──────────────────
  低影響力  | 初階客戶成功     | 初階客戶成功
            | 經理 #2（支援    | 經理 #1（支援
            | 規模化帳戶）     | 資深經理）
            | [第 4 個月聘]    | [第 2 個月聘]
  ──────────|──────────────────|──────────────────
            | 難找              | 容易找
```

*「第一個招的人：資深客戶成功經理。他們的適應期最長，而且我需要他們幫忙面試和帶領團隊的其他人。第二個招的人：一位初階客戶成功經理來支援資深的，並開始服務帳戶。第三：客戶成功營運來建立基礎設施。第四：第二位初階客戶成功經理來擴展。」*

*「對每一次招聘，我會定義標準並加權。對資深客戶成功經理，我會把續約管理經驗的權重設為高，文化適配度設為高，產業經驗設為中等。我不會過度重視產業經驗，因為強大的客戶成功經理能學習新產業。但續約管理技能很難教。」*

**問題：「你如何處理兩位團隊成員之間的衝突？」**

*「我會從診斷開始，而不是介入。我的假設是衝突是關於一個具體的工作問題，而不是個性衝突。大多數職場衝突都有結構性的根本原因：不清楚的權責、衝突的優先順序、或不一致的誘因。」*

*「我會先分別和每個人會面。我會問開放式問題：『帶我走過你最近感到挫折的地方。』我會聽他們是在講一個具體的專案、溝通方式、還是更深層的東西。」*

*「如果是權責不清楚，我會修正結構。定義誰負責什麼，讓團隊看得到。如果是優先順序衝突，我會讓他們在最重要的工作上達成一致，以及為什麼。如果在排除結構性原因之後確實是人際問題，那我會把他們帶到一起，用明確的規則做一次協調對話。」*

*「管理者能做的最糟糕的事就是忽視它，希望它自己解決。那是藉口小子的做法。第二糟糕的事是在不了解根本原因的情況下選邊站。那是衝衝衝的做法。正確的做法是先診斷，再行動。」*

## 如何練習

閱讀這份指南是第一步。練習才是改變發生的地方。以下是在下一次面試前演練這些工具的方法。

### 每天 15 分鐘的演練

從下面的清單中挑一個面試問題。設一個 2 分鐘的計時器。用四步驟框架大聲回答。用手機錄音。

聽回放。問自己：
- 我有在回答前釐清問題嗎？（步驟一）
- 我有找到根本原因，還是直接跳到解決方案？（步驟二）
- 我有展示多個選項嗎？（步驟三）
- 我有分享具體數字和學到的教訓嗎？（步驟四）

如果你漏了一個步驟，再回答同一個問題一次。這次，四個步驟都要到位。

### 按工具分類的練習問題

**分解樹練習：**
- 你會如何把客訴減少 50%？
- 你會如何把電子報從 1,000 訂閱者增長到 10,000？
- 你會如何把一個部門的成本削減 20%？

**是非樹練習：**
- 你的團隊上個月的產出下降了。帶我走過你會如何診斷。
- 一個產品功能的採用率低。你會如何找出原因？
- 一個區域的業績下降了。帶我走過你的分析。

**假設樹練習：**
- 一個競爭對手推出了類似的產品。你會怎麼做？
- 員工流動率上升了。你的假設是什麼，你會如何測試？
- 客戶獲取成本翻倍了。發生了什麼事？

### 模擬面試方法

找一個朋友或同事。給他們這份指南中的 5 個問題。讓他們面試你 30 分鐘。只請他們針對一件事給回饋：「我的回答有結構，還是我在漫談？」

如果你沒有練習夥伴，用你手機的錄音功能。大聲回答（不是在腦海中想）才是建立習慣的方式。想結構化答案和說結構化答案是兩種不同的技能。

### 記分卡

每次練習後，為自己打分：

```
  練習記分卡
  ===========

  我有在回答前釐清嗎？              是 / 否
  我有找到根本原因嗎？              是 / 否
  我有展示 2 個以上的選項嗎？        是 / 否
  我有使用具體數字嗎？              是 / 否
  我有分享學到的教訓嗎？            是 / 否
  我有控制在 2 分鐘以內嗎？          是 / 否

  分數：___/6

  目標：在面試前穩定達到 5/6。
```

## 最後一件事

最好的求職者不會背答案。**他們學會如何思考。**然後每一個答案聽起來都自然、有結構、有信心，因為它確實如此。

這些工具不是技巧。它們是**思考習慣。**在準備面試時練習。在工作中解決問題時練習。在個人生活中做決定時練習。

渡邊健介寫《解決問題最簡單的方法》是為了孩子。它成為日本商業類暢銷書第一名的事實告訴你一件事。**簡單的思考工具，持續應用，比任何複雜的策略都更有效。**

```
  你的面試小抄
  ==============

  回答之前：       「讓我確認我理解……」
  診斷時：         「我把它拆成了……」
  提出建議時：     「我的假設是……以下是我會如何測試。」
  解釋原因時：     「我根據三個標準來評估。」
  分享結果時：     用數字。一定要用數字。
  收尾時：         「以下是我學到的，以及我下次會做得不同的地方。」
```

從簡單開始。結構化思考。拿到錄取。
